# 2FA (TOTP) no login do WordPress — Design

**Data:** 2026-06-25
**Status:** Aprovado (aguardando review do spec escrito)
**Versão alvo:** v1.12.0 (minor, sem migração — DB v10 mantido)

## Contexto

O painel-pais (`/painel-pais`) é servido por `GuardKids\Ui\ParentApp`, que exige
`is_user_logged_in()` + guardião (`GuardianAuth::isCollaboratorOrAbove`); caso
contrário faz `auth_redirect()` para o **wp-login.php**. Ou seja: **o login do
painel é o login do WordPress**. A API REST (`guardkids/v1`) autentica por nonce
+ cookie (`requireAdmin` → `GuardianAuth::isAdmin`).

Já existe a seção **Segurança** no painel (onde mora o PIN dos pais — `ChildPin`
+ `SecurityController`). A 2FA é a próxima slice dessa seção.

## Decisões (todas confirmadas com o usuário)

1. **Ponto de enforcement:** no **wp-login.php** (momento real de autenticação).
   Protege também o wp-admin, não só o `/painel-pais`.
2. **Fator:** **TOTP** (RFC 6238), app autenticador. Sem dependência de entrega,
   funciona offline, padrão de mercado.
3. **Recuperação:** **códigos de uso único** (~10), gerados na ativação,
   hasheados, consumidos um a um.
4. **Escopo de conta:** 2FA é **por usuário WP** (cada guardião tem o próprio
   app), diferente do PIN que é global por conta. Armazenamento em `user_meta`.
5. **Segredo TOTP em claro** no `user_meta` no MVP (mesmo padrão do plugin
   oficial *Two-Factor*). Cifrar com salts do WP fica como hardening futuro.
6. **Lib JS de QR** no front (ex.: `qrcode`) para renderizar o QR a partir do
   `otpauthUri`, com a chave manual como fallback.
7. **Desativar 2FA exige um código válido** (TOTP ou recuperação) — não basta a
   sessão aberta.
8. **Sem premium gate** — segurança não é paywall.

## Arquitetura

Abordagem: **TOTP artesanal em PHP puro + tela intermediária no wp-login**,
modelada no fluxo consagrado do plugin oficial *Two-Factor*. Zero dependências
novas no backend (front ganha só a lib de QR).

### Componentes (unidades isoladas)

| Unidade | Tipo | Responsabilidade | Depende de |
|---|---|---|---|
| `Totp` | puro | `generateSecret()`, `provisioningUri()`, `verify()` (base32 + HMAC-SHA1, janela ±1) | nada (hash/random nativos) |
| `RecoveryCodes` | puro | `generate(n=10)`, `hashAll()`, `verifyAndConsume(code, hashes)` | `password_hash`/`password_verify` |
| `TwoFactorStore` | acoplado ao WP | lê/grava as `user_meta` da 2FA | `get_user_meta`/`update_user_meta`/`delete_user_meta` |
| `TwoFactorController` | REST | setup/activate/status/disable/regenerate (sempre sobre o user logado) | `Totp`, `RecoveryCodes`, `TwoFactorStore`, `RateLimiter` |
| `TwoFactorLogin` | hooks | interstício no wp-login (`authenticate` + `login_form_*`) | `Totp`, `RecoveryCodes`, `TwoFactorStore`, `RateLimiter` |

`includes/Security/` para `Totp`, `RecoveryCodes`, `TwoFactorStore`,
`TwoFactorLogin`; `api/Controllers/` para `TwoFactorController`.

### Armazenamento (`user_meta` no usuário WP)

| Meta key | Conteúdo |
|---|---|
| `guardkids_2fa_enabled` | bool — 2FA ativa |
| `guardkids_2fa_secret` | segredo base32 do TOTP (em claro no MVP) |
| `guardkids_2fa_pending_secret` | segredo provisório durante a ativação (antes do 1º código confirmar) |
| `guardkids_2fa_recovery` | array de `password_hash()` dos códigos (somem conforme usados) |
| `guardkids_2fa_login_nonce` | hash + expiração (~10 min) do token do interstício |

Sem schema novo → **nenhuma migração**, `GUARDKIDS_DB_VERSION` segue 10.

## Fluxo de ativação (painel-pais)

Novas rotas REST sob `guardkids/v1/security/2fa` (auth = nonce +
`requireAdmin`, agem sobre o **usuário logado atual**):

| Método/rota | Body | Resposta | Notas |
|---|---|---|---|
| `GET /security/2fa` | — | `{ enabled, recoveryRemaining }` | status |
| `POST /security/2fa/setup` | — | `{ secret, otpauthUri }` | gera segredo **pendente** (não ativa ainda) |
| `POST /security/2fa/activate` | `{ code }` | `{ enabled: true, recoveryCodes: [...] }` | confere o código contra o segredo pendente; promove a enabled; devolve os códigos de recuperação **uma única vez** |
| `POST /security/2fa/recovery-codes` | `{ code }` | `{ recoveryCodes: [...] }` | regenera (exige código válido) |
| `DELETE /security/2fa` | `{ code }` | `{ enabled: false }` | desativa (exige código válido) |

`provisioningUri` = `otpauth://totp/<issuer>:<label>?secret=<base32>&issuer=<issuer>`,
com `issuer` = nome do site e `label` = e-mail/login do usuário.

## Fluxo de enforcement (wp-login.php) — o coração

Modelado no padrão do plugin oficial *Two-Factor*:

1. **Filtro `authenticate`** (prioridade ~30, depois de
   `wp_authenticate_username_password` @20): se `$user` é `WP_User` válido **e**
   tem 2FA ativa, **não completa o login**:
   - gera `login_nonce` (hash + expiração, gravado em `user_meta`);
   - `wp_clear_auth_cookie()`;
   - renderiza a **tela intermediária** com o chrome do wp-login
     (`login_header`/`login_footer`): input do código + hidden `wp-auth-id`
     (user id), `wp-auth-nonce` (login_nonce), `redirect_to`;
   - `exit`.
2. **Hook `login_form_validate_2fa`** (ação `login_form_{action}`) recebe o POST:
   - valida o `wp-auth-nonce` contra o `login_nonce` gravado do `wp-auth-id`
     (e a expiração);
   - confere **TOTP** (janela ±1) **ou** código de recuperação (consumindo-o),
     com rate limit;
   - sucesso → `wp_set_auth_cookie($user_id)`, apaga o `login_nonce`, redireciona
     pro `redirect_to`/admin;
   - falha → re-renderiza a tela com erro (rate-limited; após N falhas, bloqueia
     temporariamente).

> O `login_nonce` impede pular a etapa da senha (só é emitido após a senha
> correta) e tem vida curta. O cookie de auth só é setado no passo 2.

## Frontend (app-parent)

- `src/api/twofactor.ts` — espelha as rotas (`getStatus`, `setup`, `activate`,
  `regenerateRecovery`, `disable`).
- `TwoFactorSection` na aba **Segurança** (ao lado do PIN), 3 estados:
  - **desativado:** texto explicativo + botão "Ativar 2FA";
  - **setup:** QR (renderizado da `otpauthUri` via lib `qrcode`) + chave manual
    copiável + input do código de confirmação;
  - **ativo:** badge "2FA ativa", botões "Regenerar códigos" e "Desativar".
- Códigos de recuperação exibidos **uma única vez** num dialog com copiar/baixar
  (segue o padrão visual do `PinDialog.tsx`).
- Disable/regenerate pedem o código atual num dialog antes de chamar a API.

## Erros e segurança

- **Fail-closed** em tudo: formato inválido, sem 2FA, nonce expirado/ausente →
  nega.
- **Rate limiting** (reusa `includes/Security/RateLimiter.php`) nas tentativas de
  código tanto no login quanto na ativação/disable.
- Headers de resposta já cobertos pelo `RestHeaders` (inclui `no-store`).
- Segredo pendente é descartado se a ativação não for confirmada.
- Recuperação: cada código serve uma vez; `recoveryRemaining` avisa quando
  estão acabando.

## Testes

- **PHPUnit (unit):**
  - `Totp` com os **vetores oficiais RFC 6238** (SHA-1) + base32 round-trip +
    janela de tempo.
  - `RecoveryCodes` (geração, hash, consumo único, rejeição de reuso).
  - `TwoFactorController` com stub de `user_meta` (setup→activate→status→disable).
  - `TwoFactorLogin`: lógica pura de validação de nonce + verificação de código
    (sem o ciclo HTTP do wp-login).
- **Vitest (app-parent):** `twofactor.ts` (chamadas/erros) + estados do
  `TwoFactorSection` (desativado/setup/ativo) e o dialog de códigos.

## Fora de escopo (YAGNI)

- Cifrar o segredo TOTP (hardening futuro).
- 2FA por e-mail ou WebAuthn/passkeys.
- 2FA obrigatória/forçada por política (fica opt-in por usuário).
- Reset por SSH documentado (a recuperação por códigos já cobre o lockout do
  pai; o reset via `wp user meta delete` continua existindo como rede final, mas
  não é entregável desta frente).
